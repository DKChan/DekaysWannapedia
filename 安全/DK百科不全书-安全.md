## 身份验证的基本原理

身份验证基于三个主要因素：你知道什么（如密码）、你拥有什么（如智能卡）以及你是什么（如生物特征）。这些因素可以单独使用，也可以组合使用以提高安全性。

1. **你知道什么**：这是最常用的身份验证方法之一，例如要求用户提供密码或PIN码。虽然这种方法相对简单易行，但也存在被猜测或窃取的风险。
2. **你拥有什么**：这种类型的身份验证依赖于物理物品，如USB密钥、智能卡或手机等。当用户需要访问系统时，必须同时拥有相应的物品和密码。
3. **你是什么**：生物识别技术通过用户的独特身体特征或行为特征来进行身份验证，比如指纹、面部识别、虹膜扫描或声纹识别等。这些方法通常比传统的密码更难以伪造。

## 多因素认证

为了增强安全性，许多组织采用多因素认证（MFA），它结合了上述至少两种不同的身份验证方法。例如，一个典型的MFA场景可能包括输入密码（你知道什么）和接收一个发送到手机上的验证码（你拥有什么）。

可以参考《零信任网络》中"用户安全"章节, 相关U2F内容

### hash算法
| 特性 | 加密哈希 (SHA-256, SHA-3, MD5 等) | 通用哈希 (SipHash, xxHash, MurmurHash 等) |
|:---|:---|:---|
| **首要目标** | 安全性：抗碰撞攻击、抗原像攻击、次原像攻击 | 性能与低冲突率：快速计算，均匀分布输入 |
| **输入敏感性** | 极高：即使输入微小变化，输出也天差地别 | 较高/中等：确保输入变化导致输出变化 |
| **输出长度** | 固定且较长（MD5: 128bit, SHA-256: 256bit） | 固定且适中（通常 64bit 或 128bit） |
| **计算速度** | 非常慢（相对通用哈希） | 非常快（通常是加密哈希的10倍以上速度） |
| **对抗恶意输入** | 核心设计目标 | 设计取舍决定（SipHash 设计目标包含，xxHash 不包含） |
| **应用场景** | 数字签名、证书、密码存储、完整性校验（如文件下载） | 哈希表、布隆过滤器、负载均衡、校验和（非安全） |

### 不同键大小下的计算性能趋势
| 算法 | 短键 (如4-16字节) | 中等键 (如100字节) | 长键 (如1KB以上) |
|:---|:---|:---|:---|
| AES-NI | 非常快 ⁸ | 非常快 (硬件流水线优化) ⁸ | 非常快 (硬件流水线优化) ⁸ |
| MurmurHash | 极快 ⁶ | **极快** (算法设计高效) ¹ ⁶ | 极快 (算法设计高效) ¹ ⁶ |
| SipHash | 较快 (有固定开销) ¹ | 较快 (性能稳定) ¹ | 较快 (性能稳定) ¹ |

### 算法特性对比
| 特性 | AES-NI (e.g., AES-based Hashing) | MurmurHash | SipHash |
|:---|:---|:---|:---|
| **性能** | 快 (依赖硬件AES-NI指令集) | 极快 (非加密哈希) | 较快 (加密安全哈希，比非加密哈希慢) |
| **安全性** | 高 (密码学安全，抗攻击) | 低 (不适用于安全场景) | 中到高 (抗哈希洪水攻击，适合密钥验证) |
| **抗碰撞性** | 极低碰撞率 (密码学强度) | 低碰撞率 (非安全场景下良好) | 低碰撞率 (设计考虑安全性) |
| **键大小影响** | 不同键大小性能稳定 (硬件优化) | 键大小影响小，非常高效 | 键大小影响较小，性能较稳定 |
| **硬件加速** | 依赖AES-NI指令集 (Intel/AMD CPU) | 无依赖，纯软件算法 | 无依赖，纯软件算法 |
| **主要应用场景** | 加密通信、数据加密、安全存储 | 哈希表、布隆过滤器、内部数据结构 | 防止哈希洪水攻击、网络协议、密钥验证 |
| **典型用例** | Linux内核加密、VPN、磁盘加密 | Redis、Hadoop、Kafka | Python、Ruby、Perl等语言的哈希表实现 |

**SipHash(1-2,1-3)** redis, rust使用, 和下面相比稍慢, 防构造冲突, 防hash dos

**AES-NI** 依赖硬件, 但速度快, 安全度高

**MurmurHash** C++, 部分java, DynamoDB分区键 ; 速度快,分布均匀

**xxHash** 早期go, rust FxHash; 极端的快, 但是不抗碰撞

**DJB2** 简单, 但是碰撞率高

**CRC16/CRC32** redis的分片
