# 工程目录结构存在问题探讨 {#工程目录结构存在问题探讨 .unnumbered}

## 背景 {#背景 .unnumbered}

在epc后, 某次规范中心的服务目录结构,服务的结构变成了

可以看到目前的工程目录结构是有点不同于DDD的Layered Architecture(DDD分层架构)或者Rhombic Symmetric Architecture(菱形对称架构), 也稍微不同于Uncle Bob的Clean Architecture(整洁架构)

## 存在问题&解决方案 {#存在问题解决方案 .unnumbered}

### repo的interface定义不合理

#### 1.1存在问题 {#存在问题 .unnumbered}

我们从上面的目录树可以看到定义抽象的接口和具体实现接口的结构体都是在同一模块下的, 那么我们分析一下他们的依赖关系, 如下图:

<https://docs.qq.com/flowchart-addon>

可以看到领域层因为import了对应repo的代码,还是会依赖于repo的;

那么有聪明的小朋友就要问了: \"我们为什么会抽象出interface呢? 嗯嗯嗯\.... 好像是因为要符合SOLID原则里的DIP, 可是这里我们并没有做到依赖反转呢!\"

是的, 因为interface这部分的抽象定义还是放在了repo中, 所以我们的领域层还是会对基础设施层有静态依赖, 那么耦合的问题并没有解决

#### 1.2解决方案 {#解决方案 .unnumbered}

##### 1.2.1 抽离interface {#抽离interface .unnumbered}

既然interface还在repo里, 那么我们将其抽离独立出来, 不就迎刃而解了吗? 这样的话, 依赖关系就变成了:

<https://docs.qq.com/flowchart-addon>

确实这样消除了对repo的直接依赖关系, 但是多出的interface抽象层模块; 增加这个模块后同样会有弊端:

1.  仅仅是为了DIP而分离出来的抽象层, 没有任何自我的领域属性, 也没有明确的边界, 和对应的领域层更是有千丝万缕的耦合关系

2.  没有明确职责的模块, 最容易发生的事情就是被污染, 如果发生被污染的情况, 那么接下来就是连logic和repo的职责变的不清晰(但这算是小概率事情了)

#####  1.2.2 interface的定义放在领域层 {#interface的定义放在领域层 .unnumbered}

将interface的定义放在领域层后, 你会发现这依赖关系图\....什么!竟然完整地实现了DIP, 一切都刚刚好, 仿佛他在那就是应该他在那, 依赖确实被反转.

依赖关系如图:

<https://docs.qq.com/flowchart-addon>

那当然是跟着interface的定义, 一起来到了领域层呀\~你没看到依赖这就样反转了吗?不然怎么是叫依赖反转呢?怎么不叫依赖消除呢?

核心原则:
• 依赖向内: 外层依赖内层，内层不依赖外层

对照上图, 可以看出我们的services和logic是承担了\"外三层\"的工作,services层做了所有用户接口层的数据转换,数据校验,鉴权等工作, 以及还完成了部分Aggregate和对象生命周期管理的行为; 而logic层则是在完成部分Aggregate和对象生命周期管理的行为之外, 完整地完成了领域层的工作;

可以看出两层的职责不是十分清晰, 但这倒不是大问题.

#### 2.2解决方案 {#解决方案-1 .unnumbered}

谜底就在谜面上, 你可以参照前面说到的两个架构进行调整;

但和前面说的一样, 我觉得这不是大问题, 让他做用户接口层+应用层的工作, 我觉得是可以接受的, 这里没必要进行太多细分, 因为很多用户接口层的工作(如鉴权, 数据校验, 路由等)在trpc都是插件处完成的.

### entity单独存在不合理

#### 3.1存在问题 {#存在问题-2 .unnumbered}

这里赘述一下, 我们都知道Services, Entity, Value Object是构建整体逻辑非常基础的三要素, 用比较通俗不太严谨的表达就是: Value Object是一堆数据集合, 而Entity是带有身份识别ID的Value Object, 而Services则是带有动作行为的Entity;

而这里我们的entity目录, 除了定义了部分Entity外, 还有部分Value Object; 我们可以知道他存在的原因是因为这部分数据结构是因为**共用**的原因独立放置;那么就有两个问题了:

1.  领域之间共同依赖的是否应该抽象出公共子领域呢?

2.  如果是repo和domain共用了数据结构, 那么请细读DIP;

#### 3.2解决方案 {#解决方案-2 .unnumbered}

参考1.2.2里说的, 领域, 抽象接口的Entity, Value Object就该就近待在他该待的地方;如果有共同依赖那么该考虑的是\"是不是有新的公共子领域\"; 如果领域之间有静态耦合, 你该考虑你的设计是不是够高内聚低耦合

## 结语 {#结语 .unnumbered}

写在最后, 其实能够做好初步的抽象层次的分离已经是很大的进步了; 再往前的工程目录结构应该是满足了SOI的原则, 但是既然已经可以做到了SOLI那就再尽可能尽善尽美做到SOLID吧.
