## 一、核心问题：LLM 辅助开发的组织级瓶颈

个人开发者用 AI 编程助手很快，但整个团队的交付速度并未等比提升。原因在于全生命周期中出现了新的摩擦：

| 问题 | 描述 |
|------|------|
| 需求歧义被放大 | 模糊的需求被快速变成代码，误解也随之规模化 |
| 代码审查压力增大 | 变更量增加，不一致性更容易引入 |
| "生成"≠"对齐" | 集成和测试问题更多暴露 |
| 生产风险更难评估 | 变更量上升时，风险推理困难 |

> **类比**：买了法拉利却在泥泞路上开 — 引擎强大，但到达时间取决于路况。

## 二、什么是 SPDD？

**Structured Prompt-Driven Development (SPDD)** 是一种将 **Prompts 视为一等交付工件** 的工程方法。

核心目标：**使 AI 生成的变更可控、可审查、可复用**，将个人效率转化为组织级能力。

### 两大核心组件

#### 1. REASONS Canvas（结构化 Prompt 模板）

7 部分结构，引导 Prompt 从 **意图 → 设计 → 执行 → 治理**：

| 维度 | 名称 | 作用 |
|------|------|------|
| **R** | Requirements | 要解决什么问题？DoD 是什么？ |
| **E** | Entities | 领域实体及关系 |
| **A** | Approach | 我们将如何满足需求的策略 |
| **S** | Structure | 变更在系统中的位置；组件和依赖 |
| **O** | Operations | 将抽象策略分解为可测试的具体实现步骤 |
| **N** | Norms | 跨领域工程规范（命名、可观测性、防错等） |
| **S** | Safeguards | 不可妥协的边界（不变式、性能限制、安全规则等） |

#### 2. SPDD 工作流

将 Prompt 纳入代码同等的管理规范：**提交历史、Code Review、质量门禁**。

**核心原则**：
> **当现实出现偏差时，先修复 Prompt，再更新代码。**

### openspdd CLI 命令

| 命令 | 类型 | 用途 |
|------|------|------|
| `/spdd-story` | 可选 | 将大需求拆分为独立用户故事（INVEST 原则） |
| `/spdd-analysis` | 核心 | 从需求中提取领域关键词，扫描代码，生成战略分析 |
| `/spdd-reasons-canvas` | 核心 | 生成完整 REASONS Canvas |
| `/spdd-generate` | 核心 | 读取 Canvas，按序生成代码 |
| `/spdd-api-test` | 可选 | 生成 cURL API 测试脚本 |
| `/spdd-prompt-update` | 核心 | 需求变更时增量更新 Canvas |
| `/spdd-sync` | 核心 | 将代码侧变更同步回 Canvas |

## 三、完整工作流示例（计费引擎增强）

### 案例背景
将静态定价模型升级为多计划、模型感知的动态计费：
- 新增 `modelId` 必填字段
- Standard Plan：沿用月度配额，超出按模型单价计算
- Premium Plan：无配额，按 prompt/completion token 分开计费

### 6 步执行流程

```
Step 1: 创建初始需求        → 用户故事（BA/PO 提供或 /spdd-story 生成）
Step 2: 澄清分析           → 明确核心逻辑、范围边界、Done 定义
Step 3: 生成分析上下文      → /spdd-analysis 提取领域关键词 + 扫描代码
Step 4: 生成结构化 Prompt   → /spdd-reasons-canvas 生成 REASONS Canvas
Step 5: 生成代码           → /spdd-generate 生成产品代码 + /spdd-api-test 验证
Step 6: 生成单元测试        → 基于测试模板生成测试 Prompt → 生成测试代码
```

### 两类代码审查变更的处理策略

| 变更类型 | 定义 | 处理策略 |
|----------|------|----------|
| **逻辑修正**（行为变更） | 涉及业务规则/逻辑不匹配，会改变系统可观察行为 | **先改 Prompt，再生成代码** |
| **重构**（仅改善代码） | 不改变可观察行为的结构/样式问题 | **先改代码，再同步回 Prompt** |

## 四、三大核心技能

SPDD 对开发者提出了新的价值方向，以下三项技能是关键：

### 1. Abstraction First（抽象优先）
> *设计先于生成*

在生成任何代码之前，必须清楚：**存在哪些对象、如何协作、边界在哪里**。否则 AI 会陷入实现细节，而整体结构崩塌。

### 2. Alignment（对齐）
> *编码前锁定意图*

明确"我们要做什么 / 不做什么"，并就标准和硬约束达成一致。否则输出快，改错慢。

### 3. Iterative Review（迭代审查）
> *将输出变为可控循环*

AI 辅助应像工程流程，而非一次性草稿。没有纪律严明的审查-迭代循环，团队会不断让模型打补丁直到方案失控，或不断重启导致成本和时间失控。

## 五、适用场景评估

| 评分 | 场景 | 说明 |
|------|------|------|
| ★★★★★ | 规模化、标准化交付 | 高重复业务逻辑，需长期可维护性 |
| ★★★★★ | 高合规要求和硬约束 | 金融核心系统、多渠道/多客户部署 |
| ★★★★☆ | 团队协作和可追溯性 | 多人交付，需端到端完整追踪 |
| ★★★★☆ | 跨切面一致性工作 | 多微服务跨语言复杂重构 |
| ★★☆☆☆ | 救火式热修复 | 速度优先于架构规范 |
| ★★☆☆☆ | 探索性 spike | 快速验证想法，非生产质量软件 |
| ★☆☆☆☆ | 上下文黑洞 | 领域定义差、业务规则不清 |
| ★☆☆☆☆ | 纯创意/视觉工作 | 由品味和美学驱动（如 UI 视觉探索） |

## 六、核心价值总结

| 收益 | 影响 | 速度 | 获得 |
|------|------|------|------|
| 确定性（Determinism） | 高 | 即时 | 将逻辑编码为精确规格，显著减少幻觉 |
| 可追溯性（Traceability） | 高 | 即时 | 每项变更可追溯到结构化 Prompt |
| 更快审查 | 高 | 短期 | 代码"抵达"时更接近团队标准 |
| 可解释性 | 中-高 | 渐进 | 意图和行为在自然语言层可见 |
| 更安全演进 | 高 | 长期 | 良好定义的边界使变更更低风险 |

## 七、核心洞察

> **"AI 时代软件开发不是模型 IQ 竞赛，而是工程师认知带宽竞赛 — 我们能多清晰地思考、定义问题、做决策。"**

> **Hamming 定律**：
> *"在科学中，如果你知道自己在做什么，就不该做它。在工程中，如果你不知道自己在做什么，就不该做它。"*
