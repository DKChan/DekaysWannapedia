#### PGSQL的存储结构和索引

pgsql不是**索引组织表**(Index-organized table), 而是**堆表**（Heap-Organized Table）

表原始数据都是按堆存储, 由多个页组成, 页存有**行指针数组**和**元组**(行数据); 每次读取按(page, row)进行查找

针对大对象的行, 有TOAST方案, 就是通过行外存储, 行内存大对象的指针

如果是有声明主键/唯一索引, 会自动创建B树索引

因为扫表是直接扫堆, 索引不需要专门使用B+树进行优化, 而且存储的值堆表的定位指针, 所以大小很小, 不会影响树高

pgsql是**可扩展的索引系统**, 可选索引:

1.  B树: 点查, 范围查

2.  Hash: 点查

3.  GiST

4.  SP-GiST

5.  GIN(倒排索引)

6.  BRIN(块范围索引)

Bloom(布隆过滤器)

### MYSQL索引的数据结构

为什么使用B+树, 优势在哪

1.  叶子节点前后链表指针, 方便遍历

2.  数据集中在叶子节点, B+树节点对应缓存的单位, 做到缓存亲和

3.  非叶子节点只存索引值, 更多索引值代表更多

4.  虽然B+树稳定查询需要logN, B树是1-logN; 但是树高等问题会有少许影响(不多)

5.  范围查询优势, 找到起点后,可以直接链表扫描; B树需要多次回溯查找索引

B+树劣势:

1.  索引值冗余

2.  维护链表关系, 单索引结构内就有类似写放大的问题

#### 节点高键 (High Key)

高键是 B+树节点中存储的一个**上界值**, 表示该节点(及其子树)中所有键值的最大边界

- 传统 B+树: 节点的键范围由父节点中的分隔键隐式界定, 需要回溯父节点才能确定边界
- 带高键的节点: 每个节点显式存储自身的上界, 无需访问父节点即可判断目标键是否在本节点范围内

作用:

1. 简化并发访问时的范围判断 — 遍历到某节点时, 直接比较高键即可知道是否需要移动到右兄弟
2. 是 Blink 树的核心组成部分, 配合右链接指针实现无锁/少锁遍历

示例: 节点包含键 [3, 5, 7], 高键=9, 表示该节点覆盖范围为 [3, 9), 查找键=8 时知道仍在本节点管辖范围

#### 溢出页 (Overflow Pages)

当单个节点(页)无法容纳所有数据时, 使用额外的页来存储溢出内容

**产生场景:**

1. 变长数据(VARCHAR, TEXT, BLOB): 单条记录的值超过页大小限制, 无法内联存储
2. 重复键过多: 同一索引键对应大量行指针, 单页放不下

**实现方式:**

- 节点中存储指向溢出页的指针, 溢出页以链表形式串联
- MySQL InnoDB: 行溢出时, 页内保留前 768 字节(或 20 字节指针), 剩余存入 off-page (BLOB page)
- PostgreSQL: 使用 TOAST (The Oversized-Attribute Storage Technique), 大字段压缩或外置到 TOAST 表

**影响:**

- 读取需要额外 I/O 跟随溢出指针
- 碎片化增加, 顺序扫描性能下降
- 节点分裂时需要同步处理溢出链

#### Blink 树 (B-link Tree)

Blink 树是 B+树的并发优化变体, 由 Lehman & Yao (1981) 提出, 是 PostgreSQL 等数据库 B 树索引的实际实现基础

**核心改进: 在 B+树基础上增加两个机制**

1. **右链接指针 (Right-link / Sibling Pointer)**: 每个节点持有指向右兄弟的指针
2. **高键 (High Key)**: 每个节点存储自身键范围的上界

**解决的问题: 并发分裂时的正确性**

传统 B+树分裂流程: 分裂子节点 → 更新父节点(插入新分隔键) — 这两步不是原子的

如果在分裂完成但父节点尚未更新时, 另一个线程从父节点下来查找:
- 传统 B+树: 可能走到旧节点, 找不到目标键 → 需要对整条路径加锁(代价高)
- Blink 树: 走到旧节点后, 比较高键发现目标键超出范围 → 沿右链接移动到新分裂出的节点 → 正确找到

**并发协议优势:**

| 维度 | 传统 B+树 | Blink 树 |
|:---|:---|:---|
| 加锁粒度 | 路径级锁或子树锁 | 单节点锁(一次只锁一个节点) |
| 分裂期间 | 阻塞其他读写 | 读操作可通过右链接绕过 |
| 死锁风险 | 自顶向下加锁可能死锁 | 单向(左→右, 上→下)无死锁 |
| 实际应用 | 教科书模型 | PostgreSQL, Oracle, SQL Server |

**查找流程:**

1. 从根节点开始, 按键值选择子节点下降
2. 到达目标层级后, 比较当前节点的高键
3. 若目标键 ≥ 高键, 沿右链接移动到右兄弟, 重复比较
4. 在正确节点中查找目标键

**插入/分裂流程:**

1. 找到目标叶子节点, 加锁
2. 若节点已满, 分裂: 创建新节点, 设置右链接和高键, 将一半键移入新节点
3. 释放叶子锁, 向父节点插入新分隔键(此时即使有延迟, 其他线程仍可通过右链接正确遍历)

#### 索引的选择性

背景: 为了提升索引性能, 将字段前缀的部分作为索引值.

索引的选择性是指，不重复的索引值（也称为基数，cardinality）和数据表的记录总数(＃T)的比值，范围从1/＃T到1之间, 选择性越高则查询效率越高

需要通过不同长度的前缀的频率进行判断

#### 索引合并策略

功能: 将多个单列索引进行组合

背景: 我们都知道都是条件and才会按最左匹配(优化可能重排, 也先不考虑)命中联合索引, 但如果只有单列的索引, 就会按最左匹配到单列索引, 后续条件不会命中;

这个算法有三个情况:

1.  OR条件的结果集取并集

2.  AND条件的结果集取交集

3.  结合1,2的条件

在优化器里进行实现

#### 索引和数据的碎片

行碎片: 数据行存在多个地方的多个片段, 更多的随机访问

行间碎片: 逻辑上顺序的页/行, 在磁盘存储不是顺序存储的

剩余空间碎片: 很多页内剩余大量空间没被利用

## 事务

### 宕机后redolog在不同阶段恢复后, 如何改写缓存,刷新磁盘和处理CDC

**binlog的写入方式和写入顺序:**

1\. binlog是在**事务提交**时写入的（在 redo log commit 之后, **注意**是事务的commit, 不是2PC的commit）

2\. 每个事务对应一个binlog事件组（**不是每条 SQL 单独写**）

3\. 写入顺序：redo log prepare→ binlog write→ redo log commit(这里的commit是事务commit里2PC的阶段) (顺序是**保证同步, 严格串行**的)

```
       │─────────────────────────────────────>│                  │

       │                  │                  │─────────────────>│

       │ 3.写入redo log   │                  │                  │

核心: 两阶段提交保证 redo log 与 binlog 一致性
• Prepare阶段: redo log写入并刷盘
• Commit阶段: binlog刷盘后, redo log标记commit
```

核心规则: **重放所有已提交的事务修改 + 回滚未提交的事务修改**

1.  事务未提交(状态\[事务: 未提交, redolog:未提交, binlog:未写, buffer pool:已写, 磁盘:未写\])

2.  刚提交事务后宕机, 已提交redolog prepare(状态\[事务: 提交, redolog:prepare未落盘, binlog:未写, buffer pool:已写, 磁盘:未写\])

3.  binlog未刷盘, 已提交事务(状态\[事务: 提交, redolog: prepare, binlog:未写, buffer pool:已写, 磁盘:未写\])

4.  已写binlog, 未提交redolog commit(状态\[事务: 提交, redolog:commit, binlog:已写, buffer pool:已写, 磁盘:未写\])

1,2,3都是回滚

4继续提交

**MySQL崩溃恢复流程** (Mermaid时序图):

```mermaid
sequenceDiagram
    participant Disk as 磁盘
    participant BP as BufferPool
    participant Redo as RedoLog
    participant Bin as Binlog
    participant Undo as UndoLog
    participant Svc as 服务

    Note over Disk,Svc: 【MySQL崩溃恢复开始】
    
    Disk->>Redo: 1. 读取Checkpoint LSN
    
    loop 重放已提交事务
        Redo->>BP: 2. 重放Redo日志修改
    end
    
    BP-->>Disk: 3. 异步刷脏页到磁盘
    
    Note over Bin,Redo: 4. 检查XA事务状态一致性
    
    alt XA事务Binlog完整
        Bin->>Redo: 事务已完整记录
        Redo->>BP: 提交XA事务
        Redo->>Disk: 写入最终COMMIT事件(若未落盘)
    else XA事务Binlog不完整
        Redo->>Undo: 需要回滚
        Undo->>BP: 回滚XA事务
    end
    
    loop 回滚未提交事务
        Undo->>BP: 5. 回滚所有未提交事务
    end
    
    Note over Disk,Svc: 【恢复完成，服务可用】
    
    BP->>Bin: 正常事务写入Binlog
    Bin->>Svc: CDC捕获变更
```

```
│  【Commit阶段】                                                 │
│   4. Server写入Binlog                                          │
│   5. 强制刷盘Binlog (保证复制不丢失)                           │
│   6. InnoDB写入Redo Log (commit状态)                           │
│   7. 返回客户端成功                                            │
│                                                                │
│  【后台异步】                                                   │
│   8. 脏页异步刷盘 (不影响事务响应)                             │
│                                                                │
├────────────────────────────────────────────────────────────────┤
│ 关键设计: Binlog作为协调者，保证Redo和Binlog一致性              │
│ 崩溃恢复: 检查Binlog完整性决定提交或回滚                        │
└────────────────────────────────────────────────────────────────┘
```

### ACID隔离级别

题外话: 严格来说一致性是应用层保证的. server层而不是存储引擎层, 为了顺口加进来的.

四个隔离级别:

1.  **读未提交**

实现方式: 几乎无隔离, 直接读取行数据

脏读/脏写, 其他事务未提交修改的数据, 也被会读取.

2.  **读提交**

实现方式:

脏写避免: **行锁,** 写时持有锁可以防止脏写问题, 但是脏读无法使用读锁解决, 因为会严重影响性能

脏读避免: 维持旧值和持锁事务的新值两个版本(**快照**)

引入不可重复读问题, 事务A中有多次读取, 事务执行期间, 有其他事务修改查询值, 导致事务中两次读取数据不一致

3.  **可重复读**

实现方式: **快照级别隔离**

如果是读提交只需要对不同的查询单独创建一个快照, 就是上文解决脏读的办法

**快照级别隔离**是使用**一个**快照运行**整个**事务

pgsql的实现: 生成一个**唯一的,单调递增的事务ID**; 所有写数据都会带上事务ID(用于回滚数据的对比)

所有**创建**和**删除**都会记录事务ID, 更新操作会转换为删除+创建

控制**可见性**规则

1.  事务开始时, 列出所有进行中的事务(**未提交/中止**), 这部分的都为不可见

2.  所有中止的事务修改为不可见

3.  大于当前事务ID的修改(更晚)都为不可见, 即便是提交状态

4.  除了上面3条以外的都是可见的

会有**幻读**的风险; 可以通过间隙锁进行规避. 后续再详述

4.  **可串行化**

最同步的, 最符合标准的, 但因为响应性能问题, 之前都不会选用; 在OLTP场景发现, 事务的操作非常简短, 可串行化执行逐渐变得可考虑

**实现方式**:

1.  单工作线程

2.  两阶段加锁(2PL)

不同锁种类:

防止幻读-**谓词锁**(字段/属性维度)性能不行

索引区间锁(next key lock): 大多使用的实现, 就是行锁+间隙锁的组合

3.  可串行化的快照隔离(SSI)

核心思想: 在快照隔离(SI)基础上, 用**乐观并发控制**检测可串行化冲突, 仅在提交时判断是否需要中止

> **基于过期的条件做决定**: 事务执行期间不阻塞, 提交时才验证是否存在冲突

**后向检查 (Backward Validation)**

- 含义: 事务T提交时, 检查T的**读集合**是否与**已提交事务**的写集合存在交集
- 检测目标: 我读到的数据是否已经被别人改过了? (rw-antidependency: 我读 → 别人写)
- 实现: PostgreSQL 通过 SIREAD Lock 记录每个事务的读集合; 当其他事务提交写入时, 标记与之冲突的 SIREAD Lock
- 触发时机: 事务提交时发现自己的读集合上存在已提交的冲突写入 → 可能需要中止

**前向检查 (Forward Validation)**

- 含义: 事务T提交时, 检查T的**写集合**是否与**仍活跃事务**的读集合存在交集
- 检测目标: 我写的数据是否会让别人正在读的快照失效? (rw-antidependency: 别人读 → 我写)
- 实现: 事务写入时, 检查目标数据上是否存在其他活跃事务持有的 SIREAD Lock; 若存在则标记冲突边
- 触发时机: 当被影响的活跃事务后续提交时, 发现自身参与了冲突结构 → 可能需要中止

**Dangerous Structure (危险结构)**

SSI 并非检测到单条 rw-antidependency 就立即中止, 而是检测**连续两条 rw-antidependency 边**形成的结构:

```
T1 --[rw]--> T2 --[rw]--> T3
```

即: T1读被T2写覆盖, T2读又被T3写覆盖。当这种 pivot 结构出现且相关事务均已提交(或正在提交)时, 中止其中一个事务以打破环路

**与悲观方案(2PL)的对比**

| 维度 | 2PL | SSI |
|:---|:---|:---|
| 策略 | 悲观, 先加锁再操作 | 乐观, 先执行后验证 |
| 阻塞 | 读写互相阻塞 | 读写不阻塞, 仅提交时可能中止 |
| 死锁 | 可能, 需要死锁检测/超时 | 不会死锁, 只有中止 |
| 适用场景 | 写冲突频繁 | 读多写少, 冲突概率低 |
